Layout

Grid

Rutnät – 12, 16 eller 24 kolumner

Används av design och gränssnitt. Förenklar framför allt layoutarbete. Placera ut block med innehåll. Oftast använder vi oss utav ett existerande grid som det i Twitter Bootstrap, det går att ändra på mycket bl.a. kolumn-antal och marginaler.

Fluid vs adaptive grids

Adaptive grid:

  • Pixelbaserade dimensioner.
  • Manuellt ändra höjder/bredder vid definierade breakpoints.
  • Vi vet alltid hur innehåller kommer att se ut.
  • Kräver mindre testning.

Fluid grid:

  • Procent baserade dimensioner.
  • Innehåll flödar om och anpassar sig till storleken på sitt förälder-element.
  • Kontrollera när designen "går sönder". Anpassa där efter.
  • Framtidssäkert.

Flexbox

Flexible box layout module

Advancerad layout med CSS

Flexbox är framtiden. Med flexbox kan vi bygga advancerade layouter mer likt det som vi kan se i riktiga applikationer. Fluid layout blir lättare. Element delar upp utrymme automatiskt, vi behöver inte sätta mått på saker och ting, det räcker med att säga "ni 5 element ska gemensamt ta upp hela bredden". Kolumner blir automatiskt samma höjd. Man kan styra om flödet av innehåller. Du behöver inte längre bry dig om source order, bestäm själv i vilket ordning dina element ska presenteras.

Flexbox historia

3 versioner:

  • display: box;
  • display: flexbox;
  • display: flex;
Flexbox har en jobbig historia. display: box; var det första versionen av flexbox och implementerades i både firefox och chrome. Sedan kom mellan versionen som snabbt skrevs om till den slutgiltiga display: flex; som nu har implementeras i Chrome 21+, Opera 12.1+ (och Opera Mobile) och senaste versionen av FF (21). IE9 får ingen kärlek och IE10 har tyvär fått den konstiga mellanversionen av flexbox.

Struktur

Mobile first stylesheets

Skriv "bas-styles" som ska laddas in i alla webbläsare (och enheter) utanför media queries.

Tänk progressive enhancement och bygg på med nya styles i media queries med min-width.

Så här gillar jag att tänka när jag ska strukturera min projekt.
h1 {
	font-size: 2.5em;
}

@media (min-width: 45em) {
	h1 {
		letter-spacing: 2;
		text-shadow: 1px 1px 1px rgba(0, 0, 0, .3);
	}
}

Dela upp dina styles i flera stylesheets baserat på breakpoints.

Mobile first style: 0-up.css, 450-up.css, 720-up.css

Also okay style: base.css, 0-449.css, 450-719.css

Se till att minifiera innan lansering för att slippa extra http-requests. Om en css-processor används är detta inte ett problem.

Optimera

Speed is still a feature

Conditional loading

Ladda endast de assets som behövs.

Ett exempel med CSS och JS.

CSS: 

@media all and (min-width: 45em) {
	body::after {
		content: "desktop";
		display: none;
	}
}
JS: 

var size = window.getComputedStyle(document.body, "::after").getPropertyValue("content");
if (size === "desktop") {
	// Ladda in fler scripts etc.
}
Sparar trafik. Kan använda sig av yepnope.js för att göra tester och ladda scripts därefter.

Lazy loading

Låt browsern ladda ned all JS, men kör det bara när det behövs.

Stora delar av JS-filen är utkommenterad, när vi behöver koden så tar vi bort kommentarerna.

Gmails Lazy Loading

<script id="lazy">
	// KOD.
	// KOD..
	// KOD...
</script>

<script>
	function lazyLoad() {
		var lazyElement = document.getElementById("lazy");
		var lazyElementBody = lazyElement.innerHTML;
		var jsCode = stripOutCommentBlock(lazyElementBody);
		eval(jsCode);
	}
</script>

<button onclick="lazyLoad()">Lazy Load</button>
Jag är inte så förtjust i denna lösning. Jag gillar inte eval då den öppnar upp för exploits. Är man säker på sin sak går det bra.

Bilder

Det finns ofta en stor datakaka att spara när det kommer till bildoptimering.

W3C har skapat en hel grupp för att adressera detta problem.

Det finns en del lösningar ute som fungerar bra, upp till projektet vilken man väljer.

CSS-content

HTML: 
<img src="image.jpg"
data-src-600px="image-600px.jpg"
data-src-800px="image-800px.jpg"
alt="">
CSS: 
@media (min-device-width:600px) {
    img[data-src-600px] {
        content: attr(data-src-600px, url);
    }
}

@media (min-device-width:800px) {
    img[data-src-800px] {
        content: attr(data-src-800px, url);
    }
}
Positivt, inget JavaScript. Negativt, hindrar inte webbläsaren från att ladda ned minst 2 olika versionerna utav bilden.
